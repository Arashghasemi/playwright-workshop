1. Why Playwright?

  • Cross-Browser: Chromium, WebKit, Firefox.
  • Auto-waiting: Eliminates flaky tests by waiting for elements to be actionable.
  • Web-First Assertions: Retries automatically until conditions are met.
  • Multi-Tab & Multi-User: Browser contexts allow isolated sessions in one test.
  • Network Interception: Easily mock APIs and modify requests.

User-First Locators

The Old Way (Legacy)


// Tied to DOM structure (brittle)
cy.get('.btn-primary.submit-btn')

// XPath madness
driver.findElement(By.xpath("//div[@id='root']//button[2]"))

Playwright (Resilient)


// Tests behave like users
await page.getByRole('button', { name: 'Submit' }).click();

await page.getByLabel('Password').fill('secret');

await page.getByText('Welcome back!').waitFor();

Must-Know Methods

  • getByRole() - The gold standard for accessibility & resilience.
  • getByText() - Great for validating messages or headings.
  • expect(locator).toBeVisible() - Core assertion with auto-retry.
  • expect(locator).toHaveText() - Exact or partial text matching.
  • page.goto('/path') - Navigates relative to baseURL.

Advanced / Niche Methods

  • page.evaluate() - Run raw JavaScript inside the browser context.
  • locator.filter({ hasText: '...' }) - Narrow down elements from a list.
  • page.waitForLoadState('networkidle') - Wait until network is quiet (use carefully!).
  • locator.scrollIntoViewIfNeeded() - Fixes tricky overlay issues.
  • expect.soft() - Fails the test at the end, allowing subsequent assertions to run.

Zero-Sleep Policy

Anti-Pattern: await page.waitForTimeout(5000);

Why it's bad: It makes tests slow (always waits 5s) and flaky (fails if it takes 5.1s).


The Fix: Auto-Waiting


// BAD
await page.click('button');
await page.waitForTimeout(2000); 
await expect(page.locator('.success')).toBeVisible();

// GOOD (Playwright polls automatically up to the timeout)
await page.click('button');
await expect(page.getByText('Success')).toBeVisible();

Hydration & toPass()

Modern frontends (React/Next) might render a button before attaching event listeners (hydration). Clicking it does nothing, causing flakiness.


// toPass() retries the entire block until it succeeds 
// Great for poorly hydrated frontends or eventual consistency
await expect(async () => {
  await page.getByRole('button', { name: 'Submit' }).click();
  
  // If this assertion fails, the click is retried!
  await expect(page.getByText('Order confirmed')).toBeVisible();
}).toPass({
  intervals: [100, 250, 500, 1000],
  timeout: 10000
});

Non-Deterministic Flows

Handling A/B tests, random popups, or cookie banners.


// Using locator.or()
const cookieBanner = page.getByRole('button', { name: 'Accept Cookies' });
const loginForm = page.getByLabel('Username');

// Wait for EITHER the cookie banner OR the login form
await expect(cookieBanner.or(loginForm)).toBeVisible();

if (await cookieBanner.isVisible()) {
  await cookieBanner.click();
}

await loginForm.fill('my_user');

API Mocking (Deterministic Tests)

Intercept network requests to test UI states without touching the real backend.


test('mocks the fruit API', async ({ page }) => {
  // Intercept the API call
  await page.route('**/api/v1/fruits', async route => {
    const json = [{ name: 'Strawberry', id: 21 }];
    await route.fulfill({ json });
  });

  await page.goto('/fruits-page');
  // UI will render Strawberry, regardless of real DB state
  await expect(page.getByText('Strawberry')).toBeVisible();
});

Page Object Model (POM)

Structure: Properties -> Getters -> Actions


import { Page, Locator, expect } from "@playwright/test"

export class Filter {
  constructor(private page: Page) {}

  // --- GETTERS (Locators) ---
  get filterButton(): Locator { return this.page.getByRole("button", { name: "Filter", exact: true }) }
  get activateFilterButton(): Locator { return this.page.getByRole("button", { name: "Filter aktivieren" }) }

  // --- ACTIONS ---
  async openFilter(): Promise {
    await expect(this.filterButton).toBeVisible();
    await this.filterButton.click();
  }

  async filterByName(name: string): Promise {
    await this.openFilter();
    await this.page.getByRole("textbox", { name: "Name" }).fill(name);
    await this.activateFilterButton.click();
  }
}

The "Hybrid" Approach

Warning: Do not Over-POM!

  • Creating a POM for every single page/component makes tests highly abstracted and hard to debug.
  • The Hybrid Way: Write standard inline tests using getByRole for simple or single-use flows.
  • Only abstract into a POM when a flow (like the Filter class or Navigation bar) is used heavily across multiple test files.

Accessibility Testing (a11y)

Integrating @axe-core/playwright for WCAG compliance.


import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

test('should not have any automatically detectable accessibility issues', async ({ page }) => {
  await page.goto('/my-feature');

  const accessibilityScanResults = await new AxeBuilder({ page })
      .withTags(['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa'])
      .analyze();

  expect(accessibilityScanResults.violations).toEqual([]);
});

5 Useful CLI Features

  • --ui : Opens the interactive UI dashboard.
  • --headed : Runs tests visibly in the browser (turns off headless).
  • --debug : Pauses execution and opens the Playwright Inspector.
  • --project="Mobile Safari" : Runs tests only for a specific project.
  • --grep "@fast" : Runs only tests tagged with @fast.

Execution Modes: Playwright runs parallel by default using workers. Use test.describe.serial() if tests must run sequentially (e.g., test 2 depends on test 1).

AI: GitHub Copilot Instructions

Define project rules in .github/copilot-instructions.md to guide AI code generation.


# Playwright Test Instructions for Copilot
- Always use `getByRole` or `getByText` as the primary locator strategy.
- Do NOT use XPath or CSS selectors unless absolutely necessary.
- NEVER use `page.waitForTimeout()`. Rely on auto-waiting.
- Use `expect().toBeVisible()` instead of checking boolean element states.
- Structure tests with Arrange-Act-Assert.

Tip: Copying a raw error trace and asking an LLM "What went wrong?" usually solves syntax issues instantly.

Playwright MCP Server

Model Context Protocol allows LLMs to natively run and control Playwright contexts.

The Downsides (Cons)

  • Token Heavy: Feeding DOM trees and visual states to LLMs consumes massive token limits quickly.
  • Cost: Running iterative tests via an LLM API gets expensive very fast.
  • Speed: Slower than running pre-compiled deterministic tests.
  • Security: Giving LLMs access to navigate web contexts requires strict sandboxing to avoid unintended operations.
